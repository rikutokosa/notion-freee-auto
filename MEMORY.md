## 1. freee API 仕様

### 税区分コード
| 用途 | コード | 名称 |
|------|--------|------|
| 売上 | 129 | 課税売上10% |
| 仕入 | 136 | 課対仕入10% |

### freee請求書API（POST /iv/invoices）
- エンドポイント: `https://api.freee.co.jp/iv/invoices`
- 必須フィールド: `company_id`, `billing_date`, `tax_entry_method`, `tax_fraction`, `withholding_tax_entry_method`, `partner_title`, `lines`
- `partner_id` または `partner_code` のどちらか一方が必須
- 明細行（type=item）では `tax_rate`（0, 8, 10）と `quantity` が必須
- `tax_code` は「取引登録の下書き保存」用のオプション項目（請求書の税計算には `tax_rate` を使う）
- `unit_price` は string型で送信する
- レスポンス形式: `{"invoice": {"id": ..., ...}}`

### freee会計API（取引登録）
- エンドポイント: `https://api.freee.co.jp/api/1/deals`
- 事業所ID: 1856949

### 作業フロー
- マイナビとBEEの取引のみ、請求書登録画面から連携して仕訳を行う
- スカウト分の支払いは請求書登録ではなく仕訳登録を行う

## 2. PCA成約管理の仕訳構成

PCAは3つの登録が必要（仕入れがない場合もある）：

| # | 種別 | 勘定科目 | 取引先 | 備考 |
|---|------|----------|--------|------|
| 1 | 売上 | 人材紹介手数料 | freee売上取引ID | 請求書登録 or 仕訳登録 |
| 2 | 仕入 | スカウト手数料 | freee支出取引ID | 集客経路がある場合のみ |
| 3 | 外注支払 | PCA仕入高 | パートナー取引先 | パートナー（外注業者）への報酬 |

## 3. Notion フィールド名

| フィールド名 | 種類 | 説明 |
|-------------|------|------|
| 決定企業 | リレーション | 企業別管理DBへのリレーション |
| 税込売上 | 数値 | **freee売上仕訳のamountに使用**（税込金額で登録する） |

| freee支出取引ID | 数値 | **仕入側の取引先ID（partner_id）** - 仕入仕訳登録時に使用 |
| パートナー取引先 | 数値 | **PCA外注業者の取引先ID** - PCA仕入高仕訳登録時に使用 |
| freee処理状態 | セレクト | 処理状態管理 |
| 請求有無 | セレクト | 請求が必要かどうか |

### 重要な注意
- 「決定企業」は以前「決定企業（DB）」だったが変更済み
- 備考には「求職者名 + 入社企業名」を入れる

## 3. 運用ルール

### スケジュール実行と通知（2026-06-27更新）
- 毎日12:00 JST（03:00 UTC）に自動転記と請求書照合、および振込アラートが実行される。
- **実行フロー**: `/api/scheduled_run` → `/api/payment_alert`
- **通知先**: メール送信から **Slack Incoming Webhook** に完全移行済み（RailwayでSMTPポートがブロックされているため）。
- 環境変数 `SLACK_WEBHOOK_URL` がRailwayに設定されている。
- `NOTIFY_TO`, `RESEND_API_KEY`, `SMTP_USER` などのメール関連変数は現在未使用。

### 自動転記
- バックグラウンドでの常時自動転記（ポーリング）は廃止、スケジュール実行または手動実行のみ。
- 環境変数 `FREEE_AUTO_STOPPED=1` をRailwayに設定済み。

### 請求書
- 件名: 「人材紹介手数料」（固定）
- 備考: 「振込手数料は貴社負担でお願いいたします。」（固定）
- 社内メモ: 「本店：CA」または「本店：PCA」
- 請求書番号: 自動採番

### スキップ処理
- 入社済 × 請求不要 → エラーにせずスキップ
- スキップログはダッシュボードに表示しない

## 6. PCA DBと本店CA DBのプロパティ型の違い

| プロパティ | 本店CA | PCA | 備考 |
|-----------|--------|-----|------|
| 仕入決済期日 | formula（スペースなし） | formula（末尾スペースあり） | get_dateで両方試す |
| 集客経路 | rollup | select | db_typeで分岐 |
| 決定企業 | relation | rich_text | db_typeで分岐 |
| 求職者 | relation | title | db_typeで分岐 |
| 担当 | rollup「担当CA」 | select「担当パートナー」 | db_typeで分岐 |
| PCA仕入高 | なし | 「受取報酬料（税込）」 formula | - |
| PCA決済期日 | なし | 「パートナー決済期日 」 formula（末尾スペース） | - |
| パートナー取引先 | なし | formula (number) | 外注業者のfreee取引先ID |
| 請求ステータスの返金 | ●返金（短期離職） | ●返金（短期離職） | 「返金発生」ではない |

### ルール検索
- get_ruleは大文字小文字無視で検索（BEE→Beeのマッチ等）
- 完全一致 → case-insensitive完全一致 → case-insensitive部分一致 の順で検索

## 7. 返金処理の仕様

### 返金時のルール
- **金額**: 税込で登録
- **メモタグ/部門/取引先**: 元の登録と同じルール（集客経路に基づく）
- **備考**: 「返金 求職者名 企業名」

### 決済期日
| 種別 | 決済期日 |
|------|----------|
| 売上マイナス（差額支払） | 辞退日の翌月末 |
| 仕入マイナス（入金） | 辞退日 + 30日 |
| PCAマイナス | 辞退日 + 30日 |

### Notionプロパティ
- 「退職日•辞退日」: date型（本店CA・PCA両方に存在）

## 9. 開発時の注意事項

1. 実装前にローカルテストを行い、エラーがないことを確認してからプッシュすること
2. 毎回デプロイ後はfreee再認証URLを自動的に案内すること
3. このMEMORY.mdを毎回作業開始時に読み込むこと
4. 過去の指示を忘れないこと（ユーザーの不満の原因）
5. ユーザーからの指示内容がプロジェクト設定（MEMORY.md・HANDOVER.md・RULEBOOK.md）と矛盾・不整合がある場合は、その都度プロジェクト設定の修正を提案すること
6. 作業完了後は必ずMEMORY.md・HANDOVER.mdをリポジトリとプロジェクト共有ファイルの両方に反映すること
