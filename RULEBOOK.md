## 1. システム概要

Notionの成約管理DBを監視し、「②経理対応待ち」ステータスのレコードを自動的にfreeeに仕訳登録するシステムです。

| コンポーネント | 役割 |
|---|---|
| `notion_client.py` | NotionのDBをクエリ・レコードのステータスを更新 |
| `rules.py` | 求人データベース・集客経路ごとの仕訳ルールを定義 |
| `processor.py` | Notionレコードを取得してfreeeに登録するメイン処理 |
| `freee_client.py` | freee APIとの通信（OAuth・取引登録・請求書・証憑） |
| `app.py` | Flask Webアプリ（ダッシュボード・仕訳アシスタント） |

## 2. Notionのステータス遷移ルール

### 本店CA成約管理DB

| 処理前ステータス | 処理内容 | 処理後ステータス | freee処理状態 |
|---|---|---|---|
| `本部確認済` | 仕訳登録（仕訳のみ） | `□承諾→freee登録済` | `仕訳成功` |
| `本部確認済` | 請求書登録（要請求） | `□承諾→freee登録済` | `請求登録成功（要仕訳転記）` |
| `●入社済` | 請求書送付（要請求のみ） | `入社→請求済` | `請求成功` |
| `●入社前辞退` | freee取引削除 | `□入社前辞退→freee更新済` | `仕訳成功` |
| `●返金（短期離職）` | マイナス仕訳登録 | `□返金→freee・請求書更新済` | `仕訳成功` |

### PCA成約管理DB

| 処理前ステータス | 処理内容 | 処理後ステータス | freee処理状態 |
|---|---|---|---|
| `本部確認済` | 仕訳登録 | `承諾→freee登録済` | `仕訳成功` |
| `入社前辞退` | freee取引削除 | `入社前辞退→freee更新済` | `仕訳成功` |
| `●返金（短期離職）` | マイナス仕訳登録 | `返金→freee・請求書更新済` | `仕訳成功` |

### freee処理状態の値

| 値 | 意味 |
|---|---|
| `処理中` | 処理開始時に一時的にセット |
| `仕訳成功` | freeeへの仕訳登録が完了 |
| `仕訳失敗` | freeeへの仕訳登録が失敗 |
| `請求登録成功（要仕訳転記）` | 請求書登録完了。freee管理画面で「取引登録」ボタンを押す必要あり |
| `請求成功` | 請求書送付が完了（●入社済ステータス） |
| `エラー` | 処理中にエラーが発生 |

## 3. 請求有無の判定ルール

**優先順位**: Notionの「請求有無」フォーミュラフィールド > `rules.py`の`needs_invoice`

1. Notionの「請求有無」フィールドが `要請求` → 請求書登録を行う
2. Notionの「請求有無」フィールドが `請求不要` → 仕訳登録のみ
3. フィールドが存在しない場合 → `rules.py`の`needs_invoice`で判定（後方互換）

## 5. freee仕訳登録の仕様

### 売上仕訳（income）

| フィールド | 設定値 |
|---|---|
| 発生日 | Notionの「入社日」 |
| 決済期日 | Notionの「売上決済期日」（なければ計算） |
| 勘定科目 | `CA売上【自社】`（PCAの場合は`PCA売上`） |
| 税区分コード | `1`（課税売上10%） |
| 部門 | `本店：CA`（PCAの場合は`本店：PCA`） |
| 備考 | `{求職者名} {入社企業名}` |
| メモタグ | 担当CA名 |

### 仕入仕訳（expense）

| フィールド | 設定値 |
|---|---|
| 発生日 | Notionの「入社日」 |
| 決済期日 | Notionの「仕入決済期日」（なければルールで計算） |
| 勘定科目 | `スカウト手数料` |
| 税区分コード | `7`（課税仕入10%） |
| 取引先 | 求人DBのルールに基づく取引先名 |
| 部門 | `本店：CA`（PCAの場合は`本店：PCA`） |

### 請求書登録（freee請求書API /iv/invoices）

| フィールド | 設定値 |
|---|---|
| 件名（subject） | `人材紹介手数料` |
| 明細1（item行） | 品目名: 求職者名、単価: 税込売上（tax_entry_method="in"固定）、数量: 1 |
| 明細2（text行） | `入社企業：{入社企業名}様` |
| 備考（invoice_note） | `振込手数料は貴社負担でお願いいたします。` |
| 社内メモ | `本店：CA` または `本店：PCA` |

## 6. 仕訳アシスタントの機能

### フォーム入力タブ

- 収入/支出・発生日・決済期日・取引先・明細行を手動入力してfreeeに直接登録
- **繰り返し登録**: チェックを入れると毎月同日発生日・同じ決済期日で複数回登録
  - 発生日は毎月同じ日付（例: 7/1 → 8/1 → 9/1…）
  - 決済期日も毎月同じ日付でずれる（例: 7/31 → 8/31 → 9/30…）
  - 月末日を指定した場合は各月の末日に自動調整（例: 1/31 → 2/28 → 3/31）
- **証憑添付**: PDF・画像・Excelファイルをfreeeに証憑としてアップロードし取引に紐付け

### AI指示タブ（自然言語）

- 自然言語でfreeeへの仕訳登録を指示
- AIがfreeeのマスタデータ（勘定科目・取引先・部門・メモタグ）を参照して仕訳を提案
- 提案内容をテーブルで確認後、「freeeに登録する」ボタンで実際に登録
- **ファイル読み取り**: PDF・画像・Excel・CSVをアップロードすると内容を読み取って仕訳を自動提案
  - 返済スケジュール表 → 毎月の元金・利息を自動で複数仕訳として提案
  - 請求書・領収書 → 金額・取引先・勘定科目を読み取って仕訳を提案
  - 登録後に証憑として自動アップロード

## 7. 手動対応が必要なケース

以下のケースは自動処理をスキップし、Notionに「要確認」として記録されます：

| ケース | 理由 |
|---|---|
| 求人DBが「都度確認」 | 決済期日が毎回異なるため |
| 求人DBが「申請フォーム」 | Hitolink等、外部フォームでの申請が必要 |
| 入社前辞退でfreee取引IDが未設定 | 削除対象の取引が特定できない |
| 返金率が0%または未設定 | マイナス仕訳の金額が計算できない |
| 入社済で請求書IDが未設定 | 送付対象の請求書が特定できない |
| 求人DBに対応ルールがない | rules.pyに未登録の求人DBの場合 |

## 8. 証憑アップロードの仕様

- エンドポイント: `POST /api/assistant/upload_receipt`
- freee APIの `/api/1/receipts` を使用
- アップロード後、取引IDが指定されている場合は `/api/1/deals/{id}/receipts` で紐付け
- 対応形式: PDF・PNG・JPG・JPEG・Excel・CSV

---

## 9. ファイル読み取りの仕様

- エンドポイント: `POST /api/assistant/extract_file`
- 対応形式とテキスト抽出方法:

| 形式 | 抽出方法 |
|---|---|
| CSV | Pythonの`csv`モジュールで直接読み取り（最大200行） |
| Excel (.xlsx/.xls) | `openpyxl`または`xlrd`で読み取り（最大3シート・300行） |
| PDF | `pdftotext`コマンドでテキスト抽出（失敗時はOpenAI Vision API） |
| 画像 (PNG/JPG) | OpenAI Vision API（gpt-4o-mini）でOCR |

- 抽出したテキストはAIのプロンプトに含めて仕訳提案に活用
- 最大8000文字まで（超過分は省略）

## 12. 振込アラートの除外ルール

振込アラート（`/api/payment_alert`）から除外される条件は以下の通り。

| 除外条件 | 内容 | 判定方法 |
|---|---|---|
| 決済済み | `payment_status: settled` | freee API自動判定 |
| 振込依頼済タグあり | 「振込依頼済」メモタグ（ID: 35285961）が付いている | freeeタグで判定 |
| CSS部門の仕訳 | CSS部門IDが含まれる場合 | 部門IDで判定 |

### 銀行連携済（IB振込データ送信済み）の扱い

- **freee APIの仕様上、「銀行連携済」は `payment_status: unsettled`（未決済）のままであり、APIで判別不可能**。
- 実際に引き落としが完了するまで `settled` にならないため、コード側での自動除外は不可能。
- **対応方法**: IB振込データ送信後、freee上で対象仕訳に「振込依頼済」メモタグを手動で付与することで、翻日以降のアラートから除外される。
