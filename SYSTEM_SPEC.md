## 3-b. 実行ログ永続保存機能

### テーブル定義（SQLite: `/data/chat_history.db`）

```sql
CREATE TABLE execution_logs (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    log_type    TEXT NOT NULL,
    executed_at TEXT NOT NULL,
    trigger     TEXT NOT NULL DEFAULT 'manual',
    summary     TEXT NOT NULL,
    detail      TEXT NOT NULL DEFAULT '{}',
    has_error   INTEGER NOT NULL DEFAULT 0
)
```

### ログ種別

| log_type | 対応機能 | 保存タイミング |
|----------|----------|-------------|
| `auto_transfer` | 自動転記 | 手動実行完了時・自動実行完了時 |
| `invoice_match` | 請求書照合 | 手動実行完了時・自動実行完了時 |
| `payment_fb` | 総合振込データ作成 | FBファイル生成時 |

### summaryフィールドのスキーマ

| log_type | キー | 内容 |
|----------|-----|------|
| `auto_transfer` | `total`, `success`, `errors`, `reviews`, `skips`, `db_type` | 処理件数・結果内訳 |
| `invoice_match` | `total_receipts`, `matched_count`, `unmatched_count`, `ai_ocr_count` | 照合結果 |
| `payment_fb` | `transfer_date`, `valid_count`, `total_amount`, `skipped_count`, `tagged_count` | 振込データ生成結果 |

### 主要関数

- `_save_execution_log(log_type, summary, detail, trigger, has_error)` — SQLiteにログを保存
- `_get_execution_logs(log_type, limit=50)` — SQLiteからログを取得

### APIエンドポイント

- `GET /api/execution_logs?log_type=auto_transfer&limit=50` — 実行ログをJSONで返す

### フロントエンド表示

- 各タブ（自動転記・請求書照合・総合振込データ作成）のルール表示の上に「📋 実行ログ」プルダウンを表示
- 各ログエントリに：実行日時・トリガー（手動/自動）・サマリー・エラー有無（赤/緑バッジ）・詳細（さらに展開可能）を表示
- ログがない場合は「実行履歴なし」と表示
- 実行完了後に自動でログを更新（`loadExecLog(tabName, true)`）

## 5. 自動転記の処理フロー

### 5-1. 対象レコード

- **本店CA DB**：`PENDING_STATUSES_HONTEN` = `["本部確認済", "●入社済", "●入社前辞退", "●返金（短期離職）"]`
- **PCA DB**：`PENDING_STATUSES_PCA` = `["本部確認済", "入社前辞退", "●返金（短期離職）"]`

### 5-2. ステータス別処理

| ステータス | 処理内容 |
|-----------|---------|
| 本部確認済 | 売上仕訳＋仕入仕訳を登録（または請求書登録）|
| ●入社済 | 請求書送付（「要請求」の場合のみ。「請求不要」はスキップ）|
| ●入社前辞退 / 入社前辞退 | freeeの既存取引を削除 |
| ●返金（短期離職）| マイナス仕訳を追加登録 |

### 5-3. 請求有無の判定

1. Notionの「請求有無」フォーミュラフィールドが「要請求」→ 請求書登録
2. フィールドが存在しない場合 → `rules.py` の `needs_invoice` で判定（後方互換）

**請求有無セレクトの自動セット：** フォーミュラ型→セレクト型移行対応として、処理時に `set_invoice_required_select()` でNotionのセレクトフィールドを自動更新する。

### 5-4. CSS求人の特殊処理

CSS求人（`payment_rule: "登録不要"`）は売上仕訳を登録しない。スカウト手数料（仕入）のみ登録する。

### 5-5. PCA成約管理の追加処理

PCA DBのレコードは通常の売上・仕入仕訳に加え、パートナー（外注業者）への支払仕訳（`PCA仕入高`）も登録する。

### 5-6. 金額処理の基本ルール

- **すべての金額は税込で処理する。**
- Notionの「税込売上」「税込集客手数料」はすでに消費税込み。
- freee請求書登録時は `tax_entry_method="in"` 固定。
- 税抜変換（×1.1や÷1.1）は `rules.py` では行わない。

### 5-7. 処理完了後のステータス更新

| 処理 | 本店CA | PCA |
|------|--------|-----|
| 仕訳登録成功 | □承諾→freee登録済 | 承諾→freee登録済 |
| 請求書登録成功 | 入社→請求済 | — |
| 入社前辞退削除 | □入社前辞退→freee更新済 | 入社前辞退→freee更新済 |
| 返金処理 | □返金→freee・請求書更新済 | 返金→freee・請求書更新済 |

### 5-8. 日次スケジュール実行

- **毎日昼12時（JST）** に自動転記（全件処理）→ 請求書照合の順に自動実行
- **トリガー**: APScheduler（BackgroundScheduler）をapp.py内に内蔵。gunicorn起動時に自動起動。
- 実行エンドポイント：`POST /api/scheduled_run`（非同期実行）、`POST /api/payment_alert`（非同期実行）
- 結果はSlackに通知（Incoming Webhook使用）
- エラー・要確認がある場合はその詳細も含めて通知
- 処理対象なし：「処理対象レコードなし」と通知

## 6. 請求書照合の処理フロー

freeeのファイルボックスにある未照合の書類を取得し、freeeの仕訳と照合する。

### 6-1. 処理フロー

1. freeeファイルボックスから未照合書類を取得
2. 書類の金額・日付・取引先をOCRで読み取り（AI-OCR使用）
3. freeeの仕訳と金額・取引先・日付で照合
4. 照合成功 → freeeの仕訳に書類を添付
5. 未照合 → 手動確認待ち

### 6-2. スコアリングロジック

| 項目 | 配点 | 内容 |
|------|------|------|
| 金額完全一致 | 50点 | 書類金額と仕訳金額が完全一致 |
| 取引先名一致 | 30点 | 部分一致でも加点 |
| 日付近接 | 20点 | issue_dateとdue_dateのうち書類日付に近い方を採用 |

### 6-3. 最低スコア閾値

- **MIN_SCORE_THRESHOLD = 40**
- 最高スコアが40点未満の場合は紐づけを行わず「未照合」とする
- 低品質な候補への誤紐づけを防止するための安全装置

### 6-4. 照合結果のfreeeリンク

- 照合結果には仕訳のfreee直リンクを表示する
- URL形式: `https://secure.freee.co.jp/deals#deal_id={deal_id}`
- 過去ログに`deal_url`がない場合も`deal_id`からフォールバック生成する

## 8. 仕訳アシスタントの仕様

### 8-1. 概要

OpenAI GPT-4oを使用したfreee操作AIエージェント。Function Callingで以下のツールを使用。

| ツール | 用途 |
|--------|------|
| `search_deals` | 仕訳検索 |
| `search_invoices` | 請求書検索 |
| `register_deal` | 仕訳登録 |
| `register_invoice` | 請求書登録 |
| `delete_deals` | 仕訳削除 |
| `list_deals` | 仕訳一覧取得 |
| `list_invoices` | 請求書一覧取得 |
| `get_deal` | 仕訳詳細取得 |
| `update_deal` | 仕訳更新 |
| `create_partner` | 取引先新規作成 |

### 8-2. 仕訳登録の必須4項目

`register_deal` を呼び出す前に以下が**すべて揃っている**ことを確認すること。1つでも欠けていたら登録せずに聞き返す。

1. **発生日**（issue_date）: YYYY-MM-DD形式
2. **取引先**（partner_name）: freeeマスタに登録されている取引先名
3. **決済期日**（due_date）: YYYY-MM-DD形式
4. **対応部門**（section_name）: freeeマスタに登録されている部門名

### 8-3. 仕訳更新のルール

- **更新手順**: search_dealsで検索 → get_dealで詳細取得 → 即座にupdate_dealを呼び出す
- **既存仕訳の勘定科目構成を維持する**: ユーザーが明示的に科目変更を指示しない限り、各明細行の勘定科目は現在のまま維持する。画像や資料に別の科目名が書いてあっても、それは内訳や参考情報であり、既存仕訳の科目を書き換える根拠にはしない
- **部門名は子部門名のみ指定する**: マスタの「親部門：子部門」のうち「子部門」部分のみをsection_nameに指定する。親部門名（CSS、本店等）は指定不可（freee APIが拒否する）
- **明細行の品目・メモタグ・セグメントタグを引き継ぐ**: 更新時は既存明細のid, item_id, tag_ids, segment_tag_idを全て保持する
- **部門が解決できない場合は現在値を維持する**

### 8-4. 確認フロー（重要）

- **テキストで「よろしいですか？」と確認を求めない**。登録・更新・削除のツールを即座に呼び出す
- フロントが確認ボタンを自動表示するので、ユーザー確認はそこで行われる
- 複数件ある場合も全件まとめて一度に処理する（一件ずつ確認しない）

### 8-5. その他の行動原則

- 取引先がマスタに存在しない場合は `create_partner` で自動作成してから登録
- 削除処理：必ず `search_deals` で対象を検索してIDを取得してから削除
- 日付解釈：「先月」「今月」「7月」などを自動解釈
- 添付ファイル（OCR済みテキスト）が含まれる場合は必ず参照・活用する
- チャット履歴はSQLiteに保存（`/data/chat_history.db`）

| `shocho` | 🧾 証憑照合 | freeeファイルボックスの未紐づけ書類を仕訳と自動照合 |
| `shiharai` | 🏦 総合振込データ作成 | freeeの未決済仕訳から全銀FBファイルを生成 |
| `status` | 🖥️ システム状態 | freee認証状態・処理件数・システム状態を表示 |

URLハッシュ（`#tab-{tabId}`）でタブを直接開くことができる。

### 12-2. 自動転記タブ

- 手動実行ボタン：「全件処理（本店CA+PCA）」「本店CAのみ」「PCAのみ」の3種類
- 自動実行は毎日昼12時（JST）に実行される
- 処理中はボタンを無効化し、完了後に再有効化する
- 処理ログは画面に表示し、実行ログ（永続）はプルダウンで確認できる

---

### 12-3. 仕訳アシスタントタブ

サブタブ：「🤖 AI指示（自然言語）」「📝 フォーム入力」の2種類。

#### AI指示サブタブ

- Ctrl+Enterでメッセージ送信
- ファイル添付対応（PDF・PNG・JPG・WEBP・Excel・CSV、複数可、最大20MB）
- AIが `register_deal` を呼び出すと登録内容をUIに表示し、「この内容でfreeeに登録する」ボタンが現れる。**ボタン押下後に実際の登録が実行される**
- AIが `delete_deals` を呼び出すと削除対象をUIに表示し、「削除を実行する」ボタンが現れる。**ボタン押下後に実際の削除が実行される**
- AIはテキストで「よろしいですか？」と確認を求めない
- 登録・削除結果にはfreee仕訳への直リンク（`https://secure.freee.co.jp/deals#deal_id={id}`）を表示する

#### フォーム入力サブタブ

- 仕訳種別（収入/支出）・発生日・決済期日・取引先・明細行（複数可）を入力
- 繰り返し登録チェックボックスで最大60回分の連続登録が可能
- 証憑ファイルをドラッグ&ドロップまたはクリックで添付可能
- 登録後に証憑ファイルをfreeeにアップロードする

### 12-4. 証憑照合タブ

- 「照合プレビュー」ボタン押下で照合候補をプレビュー表示
- プレビュー確認後に「紐づけを実行」ボタンで実際の紐づけを実行
- 「リセット」ボタンでプレビューを破棄して初期状態に戻る
- 照合結果（照合済み・未照合とも）にfreee仕訳への直リンクを表示する
  - URL形式: `https://secure.freee.co.jp/deals#deal_id={deal_id}`

### 12-5. 総合振込データ作成タブ

#### プレビュー表示ルール（重要）

- 「支払プレビュー」ボタン押下でプレビューを表示
- **アラート（要確認）は常にプレビューの一番上に表示する**（振込対象グループより前）
- **同一支払期日・同一取引先の仕訳は1行にまとめて表示する**（金額は合算、件数バッジを表示）
- 振込対象は支払期日ごとにグループ化して表示する
- 各グループに「⬇ {期日}分の振込データをダウンロード」ボタンを表示する
- アラート各行には freee 仕訳への直リンク（「🔗 freeeで確認」）を表示する
  - URL形式: `https://secure.freee.co.jp/deals#deal_id={deal_id}`
- 振込対象が0件でもアラートがあれば表示する

#### FBファイル生成

- ダウンロードボタン押下でその期日分の全仕訳IDをまとめてFBファイル生成APIに送信する
- まとめ表示された場合も、まとめ前の全仕訳IDを送信する（`deal_ids`配列を使用）
- 生成後は対象仕訳に「振込依頼済」タグ（ID: 35285961）を自動付与する

### 12-6. 実行ログ（全タブ共通）

- 各タブ（自動転記・証憑照合・総合振込）に「📋 実行ログ」プルダウンを表示
- 各ログエントリ：実行日時・トリガー（手動/自動）・サマリー・エラー有無（赤/緑バッジ）
- 「🔄 更新」ボタンで最新ログを再取得できる
- 処理完了後に自動でログを更新する
