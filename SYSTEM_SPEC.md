## 3-b. 実行ログ永続保存機能

### テーブル定義（SQLite: `/data/app.db`）

```sql
CREATE TABLE execution_logs (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    log_type    TEXT NOT NULL,
    executed_at TEXT NOT NULL,
    trigger     TEXT NOT NULL DEFAULT 'manual',
    summary     TEXT NOT NULL,
    detail      TEXT NOT NULL DEFAULT '{}',
    has_error   INTEGER NOT NULL DEFAULT 0
)
```

### ログ種別

| log_type | 対応機能 | 保存タイミング |
|----------|----------|-------------|
| `auto_transfer` | 自動転記 | 手動実行完了時・自動実行完了時 |
| `invoice_match` | 請求書照合 | 手動実行完了時・自動実行完了時 |
| `payment_fb` | 総合振込データ作成 | FBファイル生成時 |

### summaryフィールドのスキーマ

| log_type | キー | 内容 |
|----------|-----|------|
| `auto_transfer` | `total`, `success`, `errors`, `reviews`, `skips`, `db_type` | 処理件数・結果内訳 |
| `invoice_match` | `total_receipts`, `matched_count`, `unmatched_count`, `ai_ocr_count` | 照合結果 |
| `payment_fb` | `transfer_date`, `valid_count`, `total_amount`, `skipped_count`, `tagged_count` | 振込データ生成結果 |

### 主要関数

- `_save_execution_log(log_type, summary, detail, trigger, has_error)` — SQLiteにログを保存
- `_get_execution_logs(log_type, limit=50)` — SQLiteからログを取得

### APIエンドポイント

- `GET /api/execution_logs?log_type=auto_transfer&limit=50` — 実行ログをJSONで返す

### フロントエンド表示

- 各タブ（自動転記・請求書照合・総合振込データ作成）のルール表示の上に「📋 実行ログ」プルダウンを表示
- 各ログエントリに：実行日時・トリガー（手動/自動）・サマリー・エラー有無（赤/緑バッジ）・詳細（さらに展開可能）を表示
- ログがない場合は「実行履歴なし」と表示
- 実行完了後に自動でログを更新（`loadExecLog(tabName, true)`）

## 5. 自動転記の処理フロー

### 5-1. 対象レコード

- **本店CA DB**：`PENDING_STATUSES_HONTEN` = `["本部確認済", "●入社済", "●入社前辞退", "●返金（短期離職）"]`
- **PCA DB**：`PENDING_STATUSES_PCA` = `["本部確認済", "入社前辞退", "●返金（短期離職）"]`

### 5-2. ステータス別処理

| ステータス | 処理内容 |
|-----------|---------|
| 本部確認済 | 売上仕訳＋仕入仕訳を登録（または請求書登録）|
| ●入社済 | 請求書送付（「要請求」の場合のみ。「請求不要」はスキップ）|
| ●入社前辞退 / 入社前辞退 | freeeの既存取引を削除 |
| ●返金（短期離職）| マイナス仕訳を追加登録 |

### 5-3. 請求有無の判定

1. Notionの「請求有無」フォーミュラフィールドが「要請求」→ 請求書登録
2. フィールドが存在しない場合 → `rules.py` の `needs_invoice` で判定（後方互換）

**請求有無セレクトの自動セット：** フォーミュラ型→セレクト型移行対応として、処理時に `set_invoice_required_select()` でNotionのセレクトフィールドを自動更新する。

### 5-4. CSS求人の特殊処理

CSS求人（`payment_rule: "登録不要"`）は売上仕訳を登録しない。スカウト手数料（仕入）のみ登録する。

### 5-5. PCA成約管理の追加処理

PCA DBのレコードは通常の売上・仕入仕訳に加え、パートナー（外注業者）への支払仕訳（`PCA仕入高`）も登録する。

### 5-6. 金額処理の基本ルール

- **すべての金額は税込で処理する。**
- Notionの「税込売上」「税込集客手数料」はすでに消費税込み。
- freee請求書登録時は `tax_entry_method="in"` 固定。
- 税抜変換（×1.1や÷1.1）は `rules.py` では行わない。

### 5-7. 処理完了後のステータス更新

| 処理 | 本店CA | PCA |
|------|--------|-----|
| 仕訳登録成功 | □承諾→freee登録済 | 承諾→freee登録済 |
| 請求書登録成功 | 入社→請求済 | — |
| 入社前辞退削除 | □入社前辞退→freee更新済 | 入社前辞退→freee更新済 |
| 返金処理 | □返金→freee・請求書更新済 | 返金→freee・請求書更新済 |

### 5-8. 日次スケジュール実行

- **毎日昼12時（JST）** に自動転記（全件処理）→ 請求書照合の順に自動実行
- 実行エンドポイント：`POST /api/scheduled_run`
- 結果は `r.kosa@bearsnavi.com` にメールで通知
- エラーあり：件名に `⚠️ [要確認]` が付く
- エラーなし：件名に `✅` が付く
- 処理対象なし：「処理対象レコードなし」と通知

## 6. 請求書照合の処理フロー

freeeのファイルボックスにある未照合の書類を取得し、freeeの仕訳と照合する。

1. freeeファイルボックスから未照合書類を取得
2. 書類の金額・日付・取引先をOCRで読み取り（AI-OCR使用）
3. freeeの仕訳と金額・取引先・日付で照合
4. 照合成功 → freeeの仕訳に書類を添付
5. 未照合 → 手動確認待ち

## 8. 仕訳アシスタントの仕様

### 8-1. 概要

OpenAI GPT-4oを使用したfreee操作AIエージェント。Function Callingで以下のツールを使用。

| ツール | 用途 |
|--------|------|
| `search_deals` | 仕訳検索 |
| `search_invoices` | 請求書検索 |
| `register_deal` | 仕訳登録 |
| `register_invoice` | 請求書登録 |
| `delete_deals` | 仕訳削除 |
| `list_deals` | 仕訳一覧取得 |
| `list_invoices` | 請求書一覧取得 |
| `get_deal` | 仕訳詳細取得 |
| `create_partner` | 取引先新規作成 |

### 8-2. 仕訳登録の必須4項目

`register_deal` を呼び出す前に以下が**すべて揃っている**ことを確認すること。1つでも欠けていたら登録せずに聞き返す。

1. **発生日**（issue_date）: YYYY-MM-DD形式
2. **取引先**（partner_name）: freeeマスタに登録されている取引先名
3. **決済期日**（due_date）: YYYY-MM-DD形式
4. **対応部門**（section_name）: freeeマスタに登録されている部門名

### 8-3. その他の行動原則

- 取引先がマスタに存在しない場合は `create_partner` で自動作成してから登録
- 削除処理：必ず `search_deals` で対象を検索してIDを取得してから削除。フロントが確認ボタンを表示するのでAIはテキストで確認を求めない
- 日付解釈：「先月」「今月」「7月」などを自動解釈
- 添付ファイル（OCR済みテキスト）が含まれる場合は必ず参照・活用する
- チャット履歴はSQLiteに保存（`/data/chat_history.db`）
