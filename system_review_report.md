# 本店経理自動化システム：全体コードレビューおよび改善提案レポート

システム全体（`app.py`, `freee_client.py`, `matcher.py`, `processor.py`, `payment.py`, `templates/`, `SYSTEM_SPEC.md`等）の通読および静的解析を実施しました。その結果、業務ロジックの不整合、フロントエンドとバックエンド間のデータ構造のミスマッチ、仕様書と実装の乖離など、複数の改善点・問題点が発見されました。

以下に、優先度順に問題点と改善案を報告します。

## 1. 優先度：高（業務ロジックのバグ・不整合）

このセクションでは、システムの正常な動作を妨げる、直ちに修正が必要な問題点について説明します。

| 項目 | 問題の概要 | 影響 | 改善案 |
| :--- | :--- | :--- | :--- |
| **PCAの「入社前辞退」ステータス判定漏れ** | `notion_client.py` ではPCAの対応待ちステータスとして `"入社前辞退"`（●なし）を取得しているが、`rules.py` の削除判定ロジックは `if current_status == "●入社前辞退":` とハードコードされている。 | PCAの入社前辞退レコードが削除処理（`delete`）に回らず、手動確認（`review`）にフォールバックしてしまう。 | `rules.py` の判定を `if current_status in ("●入社前辞退", "入社前辞退"):` に修正する。 |
| **求人媒体名のタイポ（求人DB vs 求人BD）** | `rules.py` において、ほとんどの求人媒体は `"type": "求人DB"` と定義されているが、`"Hitolink"` のみ `"type": "求人BD"` とタイポされている。 | 通常の売上仕訳では `求人DB` で判定するためHitolinkに取引先がセットされず、返金マイナス仕訳では `求人BD` で判定しているためHitolinkだけ取引先がセットされるという矛盾が生じている。 | `"Hitolink"` の `"type"` を `"求人DB"` に修正し、返金マイナス仕訳の判定も `求人DB` に統一する。 |
| **支払管理タブとAPIレスポンスの構造ミスマッチ** | `index.html` の支払プレビュー描画処理は、APIレスポンスが `{ groups: [...], alerts: [...] }` の形式であることを期待しているが、実際のAPIは `{ transfer_targets: [...], alert_targets: [...], excluded: [...] }` を返している。 | 支払プレビュー画面にデータが正しく表示されない。 | `app.py` のエンドポイント内で、`transfer_targets` を `due_date` ごとにグループ化し、フロントエンドが期待する構造に変換して返すように修正する。 |

## 2. 優先度：中（データ欠落・API呼び出しの不整合）

このセクションでは、特定の条件下でデータが欠落したり、API呼び出しが失敗したりする可能性のある問題点について説明します。

| 項目 | 問題の概要 | 影響 | 改善案 |
| :--- | :--- | :--- | :--- |
| **仕訳登録時の部門情報（section_name）欠落リスク** | `freee_client.py` の `create_deal()` では、部門IDの解決を仕訳全体（`entry`）の `section_name` からのみ行っている。 | AIが明細行レベルでのみ `section_name` を指定した場合、部門が解決されず、部門なしで登録されるリスクがある。 | `create_deal()` 内で、明細行ごとの部門解決処理を `section_name = d.get("section_name") or entry.get("section_name")` のようにフォールバックさせる。 |
| **「振込依頼済」タグ付与時のAPIリクエストボディ不整合** | `freee_client.py` の `update_deal()` はフラットなJSONボディを使用しているが、`add_furikomi_tag()` では `"deal"` キーでネストしたボディを送信している。 | freee APIの仕様によっては、どちらかがエラーになるか無視される可能性がある。 | freee APIリファレンスを確認し、正しいPUTリクエストの形式（通常はネストした形式）に統一する。 |
| **DBクエリ失敗時のサイレントフェイラー** | `notion_client.py` で本店DBとPCA DBのクエリを `try...except` で囲んでいるが、片方がエラーになってもログ出力のみで処理を続行する。 | 片方のDB取得に失敗した場合でも「正常終了」として扱われ、ユーザーが取得漏れに気づきにくい。 | クエリ失敗時には処理全体をエラーとして中断するか、エラー情報をUIに明示的に表示する。 |

## 3. 優先度：低（仕様書・ドキュメントの乖離）

このセクションでは、システムの動作に直接影響はないものの、保守性やセキュリティの観点から改善が望ましい問題点について説明します。

| 項目 | 問題の概要 | 影響 | 改善案 |
| :--- | :--- | :--- | :--- |
| **SYSTEM_SPEC.md の内容が古い** | 「単一の真実の源」と定義されている仕様書が、実際のコード（DBのパス、FBファイルの依頼人情報、ルールブックのファイル名など）と多数乖離している。 | 開発者や他のAIがシステムを理解する際の妨げになる。 | `SYSTEM_SPEC.md` および関連ドキュメントの記載内容を、現在のコードの仕様に合わせて最新化する。 |
| **ハードコードされたトークンやID** | `notion_client.py` に、`NOTION_DB_ID_HONTEN` や `NOTION_DB_ID_PCA` のデフォルト値として本番環境と思われるUUIDがハードコードされている。 | セキュリティリスクがあり、環境移行の妨げになる。 | デフォルト値を空文字にするかダミー値に変更し、必ず環境変数から取得する運用を徹底する。 |

## 総括

全体として、各機能のモジュール化は進んでいますが、**フロントエンドとバックエンドのインターフェースの不一致（支払管理）** と **業務ルールのタイポ・条件漏れ（求人BD、PCAの入社前辞退）** が、実運用において直ちに影響を及ぼすクリティカルな問題です。

まずは優先度「高」の3点（PCAの入社前辞退ステータス判定漏れ、求人媒体名のタイポ、支払管理タブとAPIレスポンスの構造ミスマッチ）の修正を行うことを強く推奨します。
