自動転記の基本フロー

Notionの本店CA・PCA成約管理DBを対象に、以下の請求ステータスのレコードを検知し、freeeに仕訳・請求書を登録します。
実行は手動のみ（ダッシュボードの「今すぐ実行」ボタンまたは仕訳プレビューから登録）。
監視対象ステータス：本店CA＝本部確認済 / ●入社済 / ●入社前辞退 / ●返金（短期離職）、 PCA＝本部確認済 / 入社前辞退 / ●返金（短期離職）。

Notionの請求ステータス 実行される処理 処理完了後のステータス freee処理状態
本部確認済 「請求有無=要請求」→ 請求書登録 + 仕入仕訳登録
「請求有無=請求不要」→ 売上仕訳 + 仕入仕訳を登録 		□承諾→freee登録済 仕訳成功 / 請求登録成功（要仕訳転記）
●入社済（要請求のみ） freee請求書IDを取得して請求書を送付 入社→請求済 請求成功
●入社前辞退 freee売上取引ID・freee仕入取引IDの仕訳を削除 □入社前辞退→freee更新済 仕訳成功
●返金（短期離職） 返金率に応じたマイナス仕訳を追加登録 □返金→freee・請求書更新済 仕訳成功
⚠️ PCA（本店PCA）の場合の追加処理

PCA成約管理DBのレコードは、売上仕訳・仕入仕訳に加えて パートナー（外注業者）への支払仕訳（PCA仕入高）も同時に登録します。 部門は「本店：PCA」固定。担当パートナーはNotionの「担当パートナー」フィールドから取得します。

💡 金額処理の基本ルール（絶対厳守）

このシステムで扱う金額はすべて税込で処理します。 Notionの「税込売上」「税込集客手数料」はすでに消費税込みの値です。 freee請求書登録時は tax_entry_method="in" 固定。 税抜変換（×1.1や÷1.1）は行いません。 「out」に変更すると二重課税になるため絶対に変更しないこと。

🏢 部門設定ルール

売上・仕入仕訳の部門は 本店CA または 本店PCA のいずれかを自動設定します。 Notionの _db_type（honten / pca）で判定し、pca の場合は「本店：PCA」、それ以外は「本店：CA」を使用します。

⚠️ 返金処理の計算ルール

返金額の計算は以下の優先順位で行います。
① Notionに「返金後入金売上」「返金後集客手数料」が入力されている場合 → その値と元の値の差額
② 未入力の場合 → 元の税抜金額 × 返金率 ÷ 100 × 1.1（税込換算）
売上返金の決済期日：退職日の翌月末。仕入返金の決済期日：退職日 + 30日。

転記実行の停止・再開ルール

操作挙動
「転記停止」ボタン メモリ内フラグ（FREEE_AUTO_STOPPED=1）を立てて停止。プロセス再起動後は環境変数の初期値に戻る。
「転記開始」ボタン 停止フラグを解除して強制開始（force=True）。
freeeトークン期限切れ 転記をスキップしてログに警告を記録。認証後に自動再開。

請求書登録の仕様

「請求有無=要請求」のレコードは仕訳登録ではなくfreee請求書APIで請求書を登録します。 売上仕訳はfreee管理画面の「取引登録」ボタンで手動転記が必要です。

請求書フィールド設定値・ルール
件名（subject）「人材紹介手数料」固定
明細1（品目行）品目名 = 求職者名（例：山田太郎様）、単価 = 税込売上、税率 = 10%
明細2（テキスト行）「入社企業：{入社企業名}様」
備考（invoice_note）「振込手数料は貴社負担でお願いいたします。」固定
社内メモ（memo）「本店：CA」または「本店：PCA」
税入力方式（tax_entry_method）「in」（税込入力）固定
請求日（billing_date）Notionの入社日
支払期日（payment_date）Notionの売上決済期日
取引先（partner_id）Notionの「freee売上取引先ID」優先、なければ取引先名で検索
🚨 請求書登録後の必須作業

請求書登録が完了すると Notion の freee処理状態が 「請求登録成功（要仕訳転記）」 になります。 この状態では売上仕訳がfreeeに登録されていません。 freee管理画面で該当請求書を開き、「取引登録」ボタンを押して売上仕訳を手動で作成してください。

📌 請求有無の判定ルール

Notionの「請求有無」フォーミュラフィールドを優先参照します（「要請求」または「請求不要」）。 フィールドが存在しない場合は集客経路ルール（RULES辞書）の needs_invoice で判定します（後方互換）。

⚠️ 請求書送付（●入社済）の条件

「●入社済」ステータスで処理が走ると、Notionの「freee請求書ID」フィールドを参照して請求書を送付します。 freee請求書IDが未入力の場合はエラーになります。 「本部確認済」の処理で請求書が登録されているか先に確認してください。

請求書照合の仕様

freeeファイルボックスの未紐づけ書類を既存仕訳と自動照合・紐づけします。

ステップ内容
① 未紐づけ書類取得 freeeファイルボックスから category=without_deal の書類を全件取得（ページング対応）
② OCR確認 freee OCRの解析結果（金額・発行日）を確認
③ AI-OCRフォールバック freee OCRで金額または発行日が取れない場合、PDFをダウンロードして OpenAI Vision API（gpt-5-mini）で再解析。最大3ページを画像化して処理。 AI-OCRでも取得できない場合は「未照合」として記録。
④ 仕訳検索 金額完全一致（許容誤差0円）・発行日 ±90日 の仕訳を検索。 売上（income）・支出（expense）の両方を対象。 発生日（issue_date）と支払期日（due_date）の両方で検索してマージ。
⑤ スコアリング 日付の近さ（最大50点：同日=50点、1日ずれ=49点…）＋ 取引先名の一致（完全一致=30点、部分一致=15点）でスコア計算。 最高スコアの仕訳1件に自動紐づけ。
⑥ 紐づけ実行 既存の紐づけ済み書類を保持したまま追加紐づけ（上書きしない）。 紐づけAPIが失敗した場合は「未照合」として記録。
⚠️ 未照合になるケース

① freee OCR・AI-OCRともに金額または発行日が取得できない
② 金額完全一致・発行日±90日の仕訳が1件も見つからない
③ 紐づけAPIが失敗した
これらのケースは手動でfreee管理画面から紐づけを行ってください。

総合振込データ作成の仕様

freeeの未決済支出仕訳から振込対象を抽出し、全銀フォーマット（FB形式）ファイルを生成します。 生成後は SBIネット銀行の総合振込画面にアップロードして使用します。

分類条件
振込対象 本店部門（本店：CA / 本店：PCA / 本店：PRA / 本店：AIスカウト / 本店：その他）の未決済支出仕訳で、 証憑（添付ファイル）ありかつ銀行口座登録済みかつ 除外条件に該当しないもの
アラート対象 ① 証憑あり・銀行口座未登録 → 「口座未登録」アラート
② 証憑なし・支払期日まで 10日以内 → 「証憑未添付」アラート
除外対象 CSS部門が1つでも含まれる仕訳、または以下の取引先・勘定科目に該当するもの

📋 FBファイル生成ルール

・形式：全国銀行協会 総合振込フォーマット（固定長120バイト、Shift-JIS）
・振込指定日：対象仕訳の中で最も早い due_date を使用
・各明細の振込日：各仕訳の due_date（未設定の場合はヘッダーの日付で代用）
・FBファイル生成確定後：対象仕訳に「振込依頼済」メモタグ（ID=35285961）を自動付与
・「振込依頼済」タグが付いている仕訳は次回以降の振込対象から除外される
・検索範囲：過去90日〜未来60日の未決済支出仕訳

仕訳アシスタントの仕様

自然言語でfreeeを操作できるAIエージェントです。OpenAI Function Callingを使用します。

機能ルール・制約
仕訳登録 AIが登録内容を提示し、ユーザーが「この内容でfreeeに登録する」ボタンを押した後に実行。 取引先がマスタに存在しない場合は自動作成してから登録。 繰り返し登録（repeat）に対応：指定月数分を連続登録。
登録必須項目 以下4項目がすべて揃っていることが登録の条件です。
  1. 発生日: YYYY-MM-DD形式の日付
  2. 取引先: freeeマスタに登録されている取引先名
  3. 決済期日: YYYY-MM-DD形式の日付
  4. 対応部門: freeeマスタに登録されている部門名
1つでも欠けている場合は登録を実行せず、不足項目を列挙してユーザーに聞き返す。
削除処理 必ず search_deals / search_invoices で対象を検索してIDを取得してから実行。 フロントエンドが削除確認ボタンを表示し、ユーザーがボタンを押した後に実際の削除が実行される。 AIはテキストで「削除しますか？」と聞かない。
照会・集計 list_deals / list_invoices / get_deal で仕訳・請求書を取得して集計・報告。 「先月の売上合計」「今月の支出一覧」などの自然言語に対応。
添付ファイル（OCR） PDF・画像をアップロードすると OpenAI Vision API（gpt-4o）でテキスト抽出。 最大20MB。抽出テキストをAIに渡して仕訳登録に活用。 「画像は読めない」とは言わない。
日付の解釈 「先月」→ 前月の年月、「今月」→ 当月の年月、「7月」→ 当年度の7月として解釈。 今年度 = 4月〜翌3月。
取引先・勘定科目の推測 マスタ一覧から最も近いものを推測して使用。 例：「ステリファイ」→「stellify」、「売上」→「売上高」、「広告費」→「広告宣伝費」
会話履歴 SQLiteに保存（Railway ボリューム /data/chat_history.db）。 セッションIDで管理。最大200件の仕訳登録ログをメモリ内に保持。
🚨 登録・削除・更新の確認ルール

登録・削除・更新はすべて必ずユーザー確認を得てから実行します。
『登録』: AIが register_deal を呼び出すと登録内容を表示。「この内容でfreeeに登録する」ボタンを押した後に実行。
『削除』: AIが delete_deals を呼び出すと削除対象を表示。「削除する」ボタンを押した後に実行。
AIは「登録しました」「削除しました」と自分から言わない。ボタン操作後に実行される。
