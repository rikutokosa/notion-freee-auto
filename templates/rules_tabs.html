自動転記の基本フロー

Notionの本店CA・PCA成約管理DBを対象に、以下の請求ステータスのレコードを検知し、freeeに仕訳・請求書を登録します。
実行方法：手動（ダッシュボードの「今すぐ実行」ボタンまたは仕訳プレビューから登録）および毎日昼12時（JST）の自動実行。自動実行結果はSlackに通知される。
監視対象ステータス：本店CA＝本部確認済 / ●入社済 / ●入社前辞退 / ●返金（短期離職）、 PCA＝本部確認済 / 入社前辞退 / ●返金（短期離職）。

Notionの請求ステータス 実行される処理 処理完了後のステータス freee処理状態
本部確認済 「請求有無=要請求」→ 請求書登録 + 仕入仕訳登録
「請求有無=請求不要」→ 売上仕訳 + 仕入仕訳を登録 		□承諾→freee登録済 仕訳成功 / 請求登録成功（要仕訳転記）
●入社済（要請求のみ） freee請求書IDを取得して請求書を送付 入社→請求済 請求成功
●入社前辞退 freee売上取引ID・freee仕入取引IDの仕訳を削除 □入社前辞退→freee更新済 仕訳成功
●返金（短期離職） 返金率に応じたマイナス仕訳を追加登録（キミナラは売上マイナスのみ・仕入側は変更しない） □返金→freee・請求書更新済 仕訳成功
⚠️ CSS求人（スカウト手数料のみ）の特例処理

集客経路の css_scout_only フラグが True の場合（CSS求人・スカウト手数料）は、仕入仕訳（スカウト手数料）のみ登録し、売上仕訳は登録しません。 処理完了後の freee処理状態は「仕訳成功」。

📌 請求有無セレクトの自動セット（移行対応）

各レコード処理時に、Notionの「請求有無」セレクトフィールドを集客経路ルールに基づいて自動セットします。 これはフォーミュラ型→セレクト型移行の後方互換処理です。セット失敗しても処理は続行されます。

⚠️ PCA（本店PCA）の場合の追加処理

PCA成約管理DBのレコードは、売上仕訳・仕入仕訳に加えて パートナー（外注業者）への支払仕訳（PCA仕入高）も同時に登録します。 部門は「本店：PCA」固定。担当パートナーはNotionの「担当パートナー」フィールドから取得します。

💡 金額処理の基本ルール（絶対厳守）

このシステムで扱う金額はすべて税込で処理します。 Notionの「税込売上」「税込集客手数料」はすでに消費税込みの値です。 freee請求書登録時は tax_entry_method="in" 固定。 税抜変換（×1.1や÷1.1）は行いません。 「out」に変更すると二重課税になるため絶対に変更しないこと。

🏢 部門設定ルール

売上・仕入仕訳の部門は 本店CA または 本店PCA のいずれかを自動設定します。 Notionの _db_type（honten / pca）で判定し、pca の場合は「本店：PCA」、それ以外は「本店：CA」を使用します。

⚠️ 返金処理の計算ルール

返金額の計算は以下の優先順位で行います。
① Notionに「返金後入金売上」「返金後集客手数料」が両方入力されている場合 → その値と元の値の差額（返金率不要）
② 未入力の場合 → 元の税抜金額 × 返金率 ÷ 100 × 1.1（税込換算）
売上返金の決済期日：退職日の翌月末。仕入返金の決済期日：退職日 + 30日。
⚠️ 返金率フィールド名：本店CA・PCA共通でNotionの「返金料率」フィールドを使用。（「返金率」フィールドは存在しない）
⚠️ キミナラ例外：キミナラは返金が発生しないため、売上マイナス仕訳のみ登録し、仕入側（集客手数料）は変更しない。

⚠️ 手動対応が必要なケース（「要確認」としてNotionに記録）
ケース理由
求人データベースが「都度確認」決済期日が毎回異なるため
求人データベースが「申請フォーム」（Hitolink等）外部フォームでの申請が必要
入社前辞退でfreee取引IDが未設定削除対象の取引が特定できない
返金率が0%または未設定（かつ「返金後入金売上」「返金後集客手数料」のどちらかが未入力）マイナス仕訳の金額が計算できない
入社済で請求書IDが未設定送付対象の請求書が特定できない
求人データベースに対応ルールがないrules.pyに未登録の求人データベースの場合
📌 日次スケジュール実行時の動作

毎日昼12時（JST）に自動転記が実行されます。実行時は「全件処理（本店CA+PCA）」と同じ処理を行います。 処理結果はSlackに通知されます。

請求書登録の仕様

「請求有無=要請求」のレコードは仕訳登録ではなくfreee請求書APIで請求書を登録します。 売上仕訳はfreee管理画面の「取引登録」ボタンで手動転記が必要です。

請求書フィールド設定値・ルール
件名（subject）「人材紹介手数料」固定
明細1（品目行）品目名 = 求職者名（例：山田太郎様）、単価 = 税込売上、税率 = 10%
明細2（テキスト行）「入社企業：{入社企業名}様」
備考（invoice_note）「振込手数料は貴社負担でお願いいたします。」固定
社内メモ（memo）「本店：CA」または「本店：PCA」
税入力方式（tax_entry_method）「in」（税込入力）固定
請求日（billing_date）Notionの入社日
支払期日（payment_date）Notionの売上決済期日
取引先（partner_id）Notionの「freee売上取引先ID」優先、なければ取引先名で検索
🚨 請求書登録後の必須作業

請求書登録が完了すると Notion の freee処理状態が 「請求登録成功（要仕訳転記）」 になります。 この状態では売上仕訳がfreeeに登録されていません。 freee管理画面で該当請求書を開き、「取引登録」ボタンを押して売上仕訳を手動で作成してください。

📌 請求有無の判定ルール

Notionの「請求有無」フォーミュラフィールドを優先参照します（「要請求」または「請求不要」）。 フィールドが存在しない場合は集客経路ルール（RULES辞書）の needs_invoice で判定します（後方互換）。

⚠️ 請求書送付（●入社済）の条件

「●入社済」ステータスで処理が走ると、Notionの「freee請求書ID」フィールドを参照して請求書を送付します。 freee請求書IDが未入力の場合はエラーになります。 「本部確認済」の処理で請求書が登録されているか先に確認してください。

請求書照合の仕様

freeeファイルボックスの未紐づけ書類を既存仕訳と自動照合・紐づけします。

ステップ内容
① 未紐づけ書類取得 freeeファイルボックスから category=without_deal の書類を全件取得（ページング対応）
② OCR確認 freee OCRの解析結果（金額・発行日）を確認
③ AI-OCRフォールバック freee OCRで金額または発行日が取れない場合、PDFをダウンロードして OpenAI Vision API（gpt-5-mini）で再解析。最大3ページを画像化して処理。 AI-OCRでも取得できない場合は「未照合」として記録。
④ 仕訳検索 金額完全一致（許容誤差0円）・発行日 ±90日 の仕訳を検索。 売上（income）・支出（expense）の両方を対象。 発生日（issue_date）と支払期日（due_date）の両方で検索してマージ。
⑤ スコアリング 金額一致（50点）＋取引先名一致（完全一致30点/部分一致15点）＋日付の近さ（最大20点）でスコア計算。
日付スコアは issue_date と due_date の両方を比較し、書類の発行日に近い方を採用。
最低スコア閾値: 40点 — 40点未満の候補は紐づけしない（誤照合防止）。
40点以上の候補のうち最高スコアの仕訳1件に自動紐づけ。
⑥ 紐づけ実行 既存の紐づけ済み書類を保持したまま追加紐づけ（上書きしない）。 紐づけAPIが失敗した場合は「未照合」として記録。
📌 実行方法・スケジュール

手動実行：ダッシュボードの「証憑照合」タブから「照合プレビュー」→「紐づけを実行」で実行できます。
自動実行：毎日昼12時（JST）に自動転記と合わせて自動実行されます。実行結果はSlackに通知されます。
対象期間：実行日から過去365日以内にfreeeファイルボックスにアップロードされた未紐づけ書類が対象です。

⚠️ 未照合になるケース

① freee OCR・AI-OCRともに金額または発行日が取得できない
② 金額完全一致・発行日±90日の仕訳が1件も見つからない
③ 候補はあるがスコアが40点未満（閾値以下）
④ 紐づけAPIが失敗した
これらのケースは手動でfreee管理画面から紐づけを行ってください。

照合結果（照合済み・未照合とも）に freee 仕訳への直リンクが表示されます。
URL形式: https://secure.freee.co.jp/deals#deal_id={id}

仕訳アシスタントの仕様

自然言語でfreeeを操作できるAIエージェントです。OpenAI Function Callingを使用します。

機能ルール・制約
仕訳登録 AIが register_deal を呼び出すとフロントが登録内容を表示。ユーザーが「この内容でfreeeに登録する」ボタンを押した後に実行。 取引先がマスタに存在しない場合は自動作成してから登録。 繰り返し登録（repeat）に対応：指定月数分を連続登録。
仕訳更新 AIが search_deals → get_deal → update_deal を呼び出して即座に更新。
更新時のルール：
・既存仕訳の勘定科目構成は、ユーザーが明示的に科目変更を指示しない限り維持する
・既存明細行の品目・メモタグ・セグメントタグを引き継ぐ
・部門名は子部門名のみ指定（解決できない場合は現在値を維持）
・画像や資料に別の科目名が書いてあっても、それは内訳・参考情報であり既存科目を書き換える根拠にしない
登録必須項目 以下4項目がすべて揃っていることが登録の条件です。
  1. 発生日: YYYY-MM-DD形式の日付
  2. 取引先: freeeマスタに登録されている取引先名
  3. 決済期日: YYYY-MM-DD形式の日付
  4. 対応部門: freeeマスタに登録されている部門名
1つでも欠けている場合は登録を実行せず、不足項目を列挙してユーザーに聞き返す。
削除処理 必ず search_deals / search_invoices で対象を検索してIDを取得してから実行。 フロントエンドが削除確認ボタンを表示し、ユーザーがボタンを押した後に実際の削除が実行される。 AIはテキストで「削除しますか？」と聞かない。
照会・集計 list_deals / list_invoices / get_deal で仕訳・請求書を取得して集計・報告。 「先月の売上合計」「今月の支出一覧」などの自然言語に対応。
添付ファイル（OCR） PDF・画像をアップロードすると OpenAI Vision API（gpt-4o）でテキスト抽出。 最大20MB。抽出テキストをAIに渡して仕訳登録に活用。 「画像は読めない」とは言わない。
日付の解釈 「先月」→ 前月の年月、「今月」→ 当月の年月、「7月」→ 当年度の7月として解釈。 今年度 = 4月〜翌3月。
取引先・勘定科目の推測 マスタ一覧から最も近いものを推測して使用。 例：「ステリファイ」→「stellify」、「売上」→「売上高」、「広告費」→「広告宣伝費」
会話履歴 SQLiteに保存（Railway ボリューム /data/chat_history.db）。 セッションIDで管理。最大200件の仕訳登録ログをメモリ内に保持。
🚨 確認フロー（重要）

AIはテキストで「よろしいですか？」と確認を求めない。即座にツールを呼び出し、フロントが確認UIを表示する。
『登録』: AIが register_deal を呼び出すと登録内容を表示。「この内容でfreeeに登録する」ボタンを押した後に実行。
『更新』: AIが update_deal を呼び出すと即座に実行される。
『削除』: AIが delete_deals を呼び出すと削除対象を表示。「削除する」ボタンを押した後に実行。
複数件ある場合はまとめて一度に処理する（一件ずつ聞かない）。

📌 429レート制限対策

OpenAI APIが429を返した場合、自動で最大3回リトライ（指数バックオフ: 2秒→4秒→8秒）。 Retry-Afterヘッダーがあればその秒数を待機。

開発・デプロイルール

ルール内容
修正フロー コード修正 → 構文チェック（python3 -c "import app"）→ GitHubにプッシュ → Railway自動デプロイ
デプロイ方式 GitHubのmainブランチにプッシュするとRailwayが自動検知して自動デプロイ
テストポリシー プッシュ前に必ずpython3 -c "import app; import processor; import notion_client; import freee_client"で構文エラーがないことを確認してから報告する
ルールブック更新 機能・ルールに変更があった場合は必ずルールブック（rules.html）も同時に更新する
SYSTEM_SPEC.md リポジトリルートの SYSTEM_SPEC.md がシステム全体の単一の真実の源（Single Source of Truth）。他のAIが引き継ぐ際はこのファイルを最初に読むこと。コード変更時は SYSTEM_SPEC.md ・ rules.html を必ず同じコミットで更新する
環境変数管理 Railway環境変数の追加・変更は、set_env_vars2.py を参考に variableCollectionUpsert GraphQL APIで行う。APIトークン・PROJECT_ID・SERVICE_ID・ENV_IDは同ファイルに記載済み。コードにハードコードしない
トークン期限切れ freeeアクセストークンは自動リフレッシュ。リフレッシュトークン切れの場合は/freee/authから再認証
日次スケジュール 毎日昼12時（JST）に自動転記・請求書照合を自動実行。実行結果（処理件数・エラー詳細）はSlackに通知。エラー・要確認がある場合はその詳細も含めて通知する。処理がない場合も『処理対象レコードなし』と通知する
スケジュール実行エンドポイント POST /api/scheduled_run — 自動転記→請求書照合の順に実行し、結果をSlack通知
Slack通知設定 Slack Incoming Webhookを使用。SLACK_WEBHOOK_URL環境変数に設定済み。RailwayはSMTPポートをブロックしているためメール通知は使用不可。
実行ログ永続保存 SQLite（/data/app.db）の execution_logs テーブルに実行ログを永続保存する。ログ種別：auto_transfer（自動転記）/ invoice_match（請求書照合）/ payment_fb（総合振込データ作成）。手動・自動どちらの実行でも保存される。
実行ログ表示 各タブ（自動転記・請求書照合・総合振込データ作成）のルール表示の上に「📋 実行ログ」プルダウンを表示。各ログに実行日時・トリガー（手動/自動）・サマリー・エラー有無バッジ・詳細を表示。API: GET /api/execution_logs?log_type=XXX&limit=50
